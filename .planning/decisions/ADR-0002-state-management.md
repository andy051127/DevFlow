# ADR-0002: 상태관리 라이브러리 선택

## 상태
확정

## 배경
Flutter에서 UI와 데이터 상태를 연결하는 상태관리 방식을 결정해야 한다.
선택에 따라 코드 구조와 테스트 용이성이 달라진다.

## 검토한 대안
| 라이브러리 | 장점 | 단점 |
|---|---|---|
| Provider | 공식 권장, 학습 곡선 낮음 | 대형 앱에서 코드 복잡도 증가 |
| Riverpod | 타입 안정성 높음, 테스트 용이, Provider 개선판 | Provider보다 초기 학습 필요 |
| Bloc | 대규모 앱에 적합, 명확한 패턴 | 보일러플레이트 과다, 소규모 앱에 과함 |

## 결정
**Riverpod** 를 선택한다.

## 이유
- Provider의 단점을 개선한 라이브러리로 타입 안정성이 높아
  런타임 오류를 컴파일 타임에 잡을 수 있음
- 레이어드 아키텍처에서 ViewModel과 자연스럽게 연결됨
- 단위 테스트 작성이 용이하여 발표 Q&A에서 테스트 설명 가능
- Bloc 대비 DevFlow 규모에 적합한 복잡도

## 결과
- 모든 상태는 Riverpod Provider로 관리
- ViewModel은 `application/view_models/` 에 위치
