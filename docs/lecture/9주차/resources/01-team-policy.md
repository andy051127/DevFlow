# 팀 구성 상세 가이드

## 1. 가능한 형태

| 형태 | 인원 | 비고 |
|---|---|---|
| 1인 팀 | 1명 | 권장 |
| 2인 팀 | 2명 | 가능, 단 조건 있음 |
| 3인 이상 | — | **허용하지 않음** |

## 2. 2인 팀의 절대 조건

1. **점수는 두 사람 모두 동일하게 부여된다.**
   - 한 명의 발표가 부진해도, 다른 한 명도 같은 점수
   - 한 명의 결과물 기여가 적어도, 점수는 같음
2. **모든 발표는 두 사람 모두 진행한다.**
   - 중간 발표: 두 명이 같은 내용을 각자 발표
   - 최종 발표: 두 명이 같은 내용을 각자 발표
3. **모든 질문은 두 사람 모두에게 던질 수 있다.**
   - "그건 짝이 했어요"는 답이 되지 않음

## 3. 짝을 고를 때 점검할 것

- 일정과 시간대가 맞는가
- 도구·플랫폼 선택에 합의가 가능한가
- 의사결정 갈등 시 조율 방식이 있는가
- 상대가 약속한 작업을 끝내지 못했을 때 본인이 떠안을 의지가 있는가

> **하나라도 불안하다면 1인 팀을 선택하세요.**
> 점수가 묶이는 것은 협업의 즐거움이자 위험입니다.

## 4. 팀 구성 변경

- 프로젝트 시작(세션 1) 종료 시점 이후로는 **변경 불가**
- 단, 한 사람이 중도 포기 시 남은 사람은 **1인 팀으로 자동 전환**
  - 이때 점수는 1인 기준으로 재산정

## 5. 팀의 권장 운영 방식

| 영역 | 권장 |
|---|---|
| 형상관리 | GitHub 단일 저장소, 브랜치 전략 합의 |
| 의사결정 로그 | `docs/decisions/ADR-NNNN-*.md` 또는 issue로 기록 |
| 진행 상황 동기화 | 주 1회 이상 짧은 동기화, AI Agent로 회의록 요약 |
| 발표 준비 | 두 명이 **동일한 슬라이드**를 각자 연습 |
