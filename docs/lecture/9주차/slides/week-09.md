# Vibe Coding Project

7주간의 AI Agent 기반 프로젝트 여정에 오신 것을 환영합니다.

---

1. 프로젝트의 정체성 — "바이브 코딩"이란 무엇인가

3. 팀 구성 정책 (1인 / 2인)
4. 평가 기준과 가산점 체계

---

## 1. 바이브 코딩(Vibe Coding)이란

> **"의도를 말하면, AI Agent가 코드를 짠다."**

- 사람은 **무엇을 만들지(What)** 와 **왜 만드는지(Why)** 에 집중
- AI Agent는 **어떻게 구현할지(How)** 를 담당
- 사람의 역할: 방향 설정, 검증, 의사결정, 그리고 **설명**

---

## 1.1 그러나 함정이 있다

AI에 맡기면 **결과는 빨리 나오지만**,
- 내가 만든 것을 **정확히 모른다**
- 질문을 받으면 **답할 수 없다**
- 문제 발생 시 **고치지 못한다**

> 이번 프로젝트가 가장 중요하게 보는 것 →
> **"AI가 만든 것을 본인이 완전히 이해하고 설명할 수 있는가"**

---

## 3. 팀 구성 정책

### 1인 팀 — 권장 옵션
- 단독 의사결정, 단독 점수
- AI Agent를 동료처럼 활용

### 2인 팀 — 가능, 단 조건 있음
- **두 사람 모두 동일한 점수를 받는다**
- 한 명이 부진하면 다른 한 명도 같이 손해
- 모든 발표는 **두 사람 모두** 진행해야 함 (동일 내용)

> **핵심 안내**: 짝을 신중히 선택하세요.
> 부담스럽다면 **1인 팀을 권장**합니다.

---

## 3.1 발표는 모두가 한다

- 중간 발표, 최종 발표 모두 **2인 팀은 두 명 다 발표**
- 같은 내용을 두 번 발표하는 형태
- 이유:
  - 한 사람이 모든 걸 알아야 한다는 강제력
  - **본인을 PR하는 훈련** — 향후 면접/이직 시 자산

## 4.1 "개발자 기본 소양"이 뭔가

발표 중·Q&A 중 다음을 묻습니다.

- 사용한 **플랫폼**은 무엇인가? 왜 그것을 선택했나?
- **앱의 구조**는 어떻게 되어 있는가? (레이어/디렉토리)
- **개발 환경 설정**은 어떻게 하는가? (처음부터)
- **빌드와 배포**는 어떤 단계로 이뤄지는가?
- **테스트**는 어떻게 작성·실행하는가?

> AI가 만들어줬어도 **본인이 답할 수 있어야** 합니다.

---

## 5. 가산점 (최대 +6점, 발표 평가에 가산)

| 항목 | 가산점 |
|---|---|
| AI Agent / 스킬 / 워크플로우 적극 활용 | **+1** |
| **본인만의 기법** (단일 md로 agent/skills/rules/commands/AGENTS.md 통합 등) 구성 + 설명 | **+2** |
| 최신 LLM Wiki 기반 **본인만의 암묵지** 관리 운영 | **+1** |
| 최신 AI Agent **리포트 발표** (10분 이상) | **+2** |

---

## 5.1 "본인만의 기법" 예시

> 단 하나의 `AUTHORING.{본인이름}.md` 파일로
> agent / skills / rules / commands / AGENTS.md 를
> **모두 부트스트랩** 하는 도구를 만든다.

- 어떤 구조로 만들었는지
- 왜 그렇게 만들었는지
- 어떻게 동작하는지
- → **말로 설명할 수 있어야** 가산점 인정

참고 사례: 워크스페이스의 `AUTHORING.DIMOHY.v1.5.3.md`

## 6. 프로젝트 주제 가이드

### 우선: **모바일 앱**
- Flutter / React Native / Android(Kotlin) / iOS(Swift) 등 자유
- 본 과정의 정체성이 "앱 프로그래밍 응용"이기 때문

### 예외: 웹 또는 데스크탑 앱
- **뚜렷한 목적**이 있을 때만 허용
- 예: 모바일로 풀 수 없는 도구, 특정 하드웨어 연동 등

---

## 7. AI Agent로 모든 것을 한다

다음 모두 **AI Agent로 자동 생성·관리**합니다.

- 기획서, 요구사항 정의
- 전체 일정 / WBS / 마일스톤
- 설계 문서, 아키텍처
- 구현, 테스트, 배포 스크립트
- 문서화, README
- **발표 자료까지**

> **단 하나의 절대 조건**:
> "내가 한 것"이 아니라 "AI가 만든 것"이라도
> **나는 모든 것을 이해하고 설명할 수 있어야 한다.**

---

## 8. 본인만의 암묵지 — LLM Wiki

- 최신 LLM Wiki(예: GitHub Copilot Wiki, Claude Docs, OpenAI Cookbook 등)를 활용
- 작업 중 알게 된 노하우를 **본인 저장소**에 정리
- 다음 프로젝트, 다음 회사에서 **자산이 됨**
- **운영하고 있음을 증명**할 수 있으면 +3점

권장 형태:
- `~/notes/` 또는 깃허브 private repo
- 카테고리: 프롬프트, 실패 사례, 패턴, 도구별 차이
