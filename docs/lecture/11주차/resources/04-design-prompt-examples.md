# 설계 프롬프트 예시

목표: 학생이 AI Agent에게 프로젝트 설계를 맡기더라도, 결과를 스스로 설명하고 검수할 수 있게 돕는다.

## 사용 방법

1. `[대괄호]` 안을 자기 프로젝트 정보로 바꾼다.
2. AI가 바로 파일을 만들지 않도록 먼저 질문과 변경 계획을 요구한다.
3. 생성된 구조를 `docs/architecture.md`, ADR, README와 연결한다.
4. 마지막에는 반드시 빌드/실행 결과와 문서를 함께 검수한다.

---

## 1. 설계 시작 전 질문을 받는 프롬프트

```text
너는 모바일 앱 아키텍트이자 수업 조교야.

우리 프로젝트 정보:
- 앱 주제: [예: 캠퍼스 분실물 공유 앱]
- 선택 플랫폼: [Flutter / React Native / Android / iOS]
- 핵심 기능 3개: [기능1, 기능2, 기능3]
- 데이터 출처: [로컬 저장 / Firebase / REST API / 미정]
- 팀 경험: [초급 / 웹 경험 있음 / 모바일 경험 있음]

요청:
1. 설계를 시작하기 전에 빠진 정보가 있으면 질문 5개를 먼저 해줘.
2. 추측한 내용은 "가정"이라고 표시해줘.
3. 답변 후 Presentation / Application / Domain / Data 기준으로 설계 방향을 제안해줘.
4. 수업 초급자가 발표할 수 있는 단어로 설명해줘.
```

---

## 2. 플랫폼 선택을 ADR로 정리하는 프롬프트

```text
우리 팀의 모바일 플랫폼 선택을 ADR로 정리해줘.

현재 선택: [Flutter / React Native / Android(Kotlin) / iOS(Swift) / 기타]
대안: [비교할 대안 1~2개]
팀 기준:
- 개발 속도: [중요/보통/낮음]
- 팀 경험: [설명]
- 목표 OS: [Android / iOS / 둘 다]
- 유지보수: [중요/보통/낮음]

산출물:
1. ADR-0001 제목
2. Context
3. Decision
4. Alternatives
5. Consequences
6. 학생이 60초 안에 말할 요약문
```

---

## 3. 레이어드 아키텍처 구조를 만드는 프롬프트

```text
아래 조건으로 우리 앱의 초기 디렉토리 구조를 제안해줘.

조건:
- 레이어: Presentation, Application, Domain, Data
- 각 레이어의 책임을 한 문장으로 설명
- 핵심 기능 3개가 어느 레이어를 지나는지 표로 정리
- 선택 플랫폼의 일반적인 파일/폴더 관례를 지켜줘

산출물:
1. 디렉토리 트리
2. 각 폴더의 책임 설명
3. 새 화면/상태/규칙/API 코드를 어디에 둘지 표로 정리
4. 구조가 너무 복잡하면 초급자용으로 줄인 대안도 제시

주의:
- 실제 파일을 만들기 전 변경 계획을 먼저 보여줘.
- 아직 확실하지 않은 정보는 질문으로 남겨줘.
```

---

## 4. `docs/architecture.md`와 Mermaid를 작성하는 프롬프트

```text
우리 프로젝트의 docs/architecture.md 초안을 작성해줘.

포함할 내용:
1. 앱 목적 한 문장
2. 전체 레이어 설명
3. 사용자 액션이 화면에서 API/DB까지 흐르는 과정
4. Mermaid flowchart 코드
5. 새 기능을 추가할 때 파일 위치를 판단하는 규칙
6. 아직 결정되지 않은 설계 이슈

Mermaid 조건:
- Presentation → Application → Domain → Data 흐름이 보여야 함
- 노드 이름은 학생이 이해하기 쉬운 단어 사용
- 너무 복잡한 다이어그램 금지
```

---

## 5. `docs/setup.md`를 작성하는 프롬프트

```text
새 팀원이 git clone 후 5분 안에 실행할 수 있도록 docs/setup.md를 작성해줘.

포함할 내용:
1. 필요한 도구와 버전
2. Windows / macOS / Linux별 설치 참고
3. clone 명령
4. 의존성 설치 명령
5. .env.example 작성 방법
6. 첫 실행 명령
7. 자주 나는 오류와 해결 방법 5개
8. 실행 성공을 확인하는 기준

주의:
- 비밀키나 실제 토큰은 문서에 쓰지 말고 placeholder로 표시
- 명령은 복사해서 실행할 수 있게 한 줄씩 작성
```

## 6. AI 결과를 검수하는 프롬프트

```text
방금 만든 구조를 학생 발표 관점에서 검수해줘.

검수 기준:
1. 새 화면 파일은 어디에 추가해야 하는가?
2. 상태/흐름 코드는 어느 폴더에 있어야 하는가?
3. 핵심 비즈니스 규칙은 어느 레이어에 있어야 하는가?
4. API/DB 코드는 어디에 있어야 하는가?
5. 현재 구조에서 잘못 배치된 것으로 의심되는 파일 3개를 찾아줘.

출력 형식:
- 학생이 60초 안에 설명할 말 5문장
- 수정이 필요한 파일/폴더 목록
- 수정 전 확인해야 할 질문
```

## 7. 나쁜 프롬프트와 좋은 프롬프트

| 나쁜 프롬프트 | 문제 | 좋은 프롬프트 방향 |
|---|---|---|
| 앱 만들어줘 | 범위와 기준이 없음 | 앱 주제, 플랫폼, 핵심 기능, 산출물 지정 |
| 구조 예쁘게 만들어줘 | "예쁘다"의 기준이 모호함 | 레이어와 파일 위치 기준 지정 |
| 문서 써줘 | 누구를 위한 문서인지 불명확 | 새 팀원이 5분 안에 실행할 setup 문서 요청 |
| 다이어그램 그려줘 | 어떤 흐름인지 불명확 | 사용자 액션 → 상태 → 규칙 → API/DB 흐름 지정 |
| 틀린 거 고쳐줘 | 검수 기준이 없음 | 새 화면/상태/규칙/API 위치 기준으로 검수 요청 |

---

## 8. 학생 자기 점검 질문

- AI가 만든 폴더 구조를 내가 60초 안에 설명할 수 있는가?
- 새 화면을 만들 때 어느 폴더에 둘지 말할 수 있는가?
- API/DB 코드는 어느 레이어에 있어야 하는가?
- `docs/architecture.md`에 현재 구조가 반영되어 있는가?
- `docs/setup.md`만 보고 다른 사람이 실행할 수 있는가?
- ADR에는 선택과 대안, 이유가 모두 들어 있는가?
